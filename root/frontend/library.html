Why I Built My Own AST Engine

Most developers interact with an Abstract Syntax Tree through an existing framework or compiler. I chose a different route. I wanted to understand how source code is transformed into a structured model that can be analyzed, queried and manipulated without relying on a black-box implementation.

The project began as a simple parser experiment but gradually evolved into a complete analysis pipeline. Source files are parsed into an AST, traversed by independent analyzers and combined into a project model that represents the architecture of an application. Reports are then generated directly from that model rather than from individual files, making the output consistent regardless of project size.

One of the primary goals was extensibility. Every analyzer focuses on a single responsibility—imports, exports, functions, classes, dependencies or future architectural rules. This modular design allows new analyses to be introduced without rewriting the existing pipeline and keeps each stage independently testable.

The long-term vision extends beyond reporting. The engine is being designed to support custom rule evaluation, architectural diagnostics and eventually automated source transformations. Instead of merely describing a codebase, it aims to explain why a problem exists and how it can be resolved.

Architecture

Designing Deterministic Systems

One concept that consistently influences my software design is determinism. Given the same inputs, a system should always produce the same output. Predictable behaviour makes debugging easier, testing more reliable and complex applications significantly easier to reason about.

This philosophy appears throughout my projects. Instead of allowing components to share mutable state freely, I prefer clear data flow, explicit state transitions and well-defined execution order. Hidden dependencies often become today's convenience and tomorrow's debugging session.

My event-driven experiments follow the same principle. Rather than storing arbitrary application state, actions are represented as immutable events that describe what happened. The current state becomes a consequence of previous events instead of a collection of unrelated variables that may change unexpectedly.

Deterministic software is rarely the fastest solution to prototype, but it usually becomes the easiest system to maintain. As projects grow, predictability becomes more valuable than clever shortcuts, and that is a trade-off I intentionally make.

Philosophy

Learning Through Building

I learn best by building software instead of collecting tutorials. Reading documentation provides the theory, but implementing an idea exposes the assumptions, mistakes and engineering trade-offs that no video can fully demonstrate.

This is why many of my projects intentionally avoid taking the shortest path. Building backend services without large frameworks, writing custom developer tooling and experimenting with compiler techniques require more effort, but they also reveal how the underlying technology actually works.

Every completed project becomes more than a finished application. It becomes a record of decisions—what worked, what failed and what should be improved in the next iteration. That process has proven far more valuable than trying to memorize APIs or follow trends.

My objective is not simply to produce applications that function today. It is to develop the engineering judgement required to design software that remains understandable, maintainable and adaptable as requirements evolve over time.
